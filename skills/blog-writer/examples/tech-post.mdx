## Rate limiting costs less than downtime

A 2024 Cloudflare report found that 67% of API outages trace back to missing or misconfigured rate limits. Not DDoS attacks. Not hardware failures. Just normal traffic from legitimate users exceeding what the server can handle.

The math is straightforward. Downtime costs somewhere between $5,600 per minute (Gartner's often-cited average) and significantly more if you process payments. A rate limiter takes a day to implement and costs essentially nothing to run.

But rate limiting isn't just about preventing outages. It's about fairness. Without limits, one heavy user can starve everyone else. Your power user running a data export shouldn't degrade the experience for thousands of normal users hitting your API.

## How token bucket actually works

Every rate limiting algorithm answers the same question: should this request go through? The differences are in how they count.

Token bucket is the most widely used algorithm, and for good reason. Imagine a bucket that holds 100 tokens. Every second, 10 new tokens drop in. Each request takes one token. When the bucket is empty, requests get rejected. When traffic is light, tokens accumulate up to the maximum - which means short bursts are fine.

This burst tolerance is what makes token bucket practical. Real traffic isn't smooth. Users click, wait, click again. APIs get hit in waves. An algorithm that expects perfectly uniform traffic will reject legitimate requests constantly.

## Three algorithms, three trade-offs

Token bucket isn't the only option. Sliding window and fixed window solve the same problem differently. The right choice depends on what matters more to your application: simplicity, precision, or burst handling.

Fixed window has a subtle problem. If your limit is 100 requests per minute and a user sends 100 requests at 11:59:59 and 100 more at 12:00:01, they've sent 200 requests in 2 seconds - twice your intended limit. Sliding window fixes this by tracking individual request timestamps, but at the cost of memory.

For most APIs, token bucket is the right starting point. It's simple, memory-efficient, and handles real-world traffic patterns gracefully. Switch to sliding window only if you need precise per-second enforcement - payment processing, billing APIs, or compliance-regulated endpoints.

<Callout variant="warning" title="Don't rate limit health checks">
Load balancers and monitoring tools hit health endpoints constantly. Rate limiting
them triggers false downtime alerts. Exclude /health and /ready from your rate
limiter, or set their limits so high they're effectively unlimited.
</Callout>
